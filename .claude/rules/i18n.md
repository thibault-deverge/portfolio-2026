# i18n conventions (next-intl 4)

- 2 locales : **`fr` (défaut)** + `en`. `localePrefix: 'as-needed'` → FR à la racine (`/`), EN préfixé (`/en`).
  `/fr/...` redirige (307) vers l'URL propre.
- Config : `i18n/routing.ts` (locales), `i18n/request.ts` (messages par requête),
  `i18n/navigation.ts` (Link/usePathname/useRouter conscients de la locale),
  `proxy.ts` (middleware — **nom Next 16**, ex-`middleware.ts`).

## Règles

- **`setRequestLocale(locale)` dans le layout ET chaque page** (sinon perte du rendu statique → page dynamique).
- **Parité des clés** : toute clé ajoutée dans `messages/fr.json` doit exister dans `messages/en.json`
  (même structure de namespaces). Ajouter dans les **deux**, jamais un seul.
- Namespaces existants : `Home`, `Nav`, `Journey`, `Work`, `Contact`. Grouper par section.
- **Aucun texte visible hardcodé** → toujours via `t('...')`.

## API

- Server Component async : `const t = await getTranslations('NS')`.
- Client Component : `const t = useTranslations('NS')` (via `NextIntlClientProvider`).
- JSX dans une traduction : `t.rich('key', { accent: (chunks) => <span className="text-accent">{chunks}</span> })`
  avec la chaîne `"... <accent>x</accent> ..."`.
- Navigation : **toujours** `Link`/`usePathname`/`redirect` depuis `@/i18n/navigation` (pas `next/link`).
